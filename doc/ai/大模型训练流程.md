# 大模型训练流程

## 概述

大模型的训练不是「把数据丢进去跑一次」那么简单。它是一个**分阶段的工程流程**：预训练（学知识）、监督微调（学对话）、对齐训练（学做人）。每个阶段的数据、目标、代价都不同。本文从数据准备到最终部署，完整走读整个训练管线。

> [!note] 前置知识
> 建议先阅读 [[大模型技术原理]] 了解 Transformer 架构，以及 [[什么是大模型]] 了解参数和 Token 的概念。

## 2. 第一阶段：预训练（Pretraining）

### 2.1 目标

让模型在海量文本上学习**语言的基础知识**：语法、语义、事实、推理模式。

```
训练任务: Next Token Prediction (预测下一个词)

输入:  "中华人民共和国的首都是"
目标:  "北京"

输入:  "if x = 5 and y = 3, then x + y ="
目标:  "8"

模型唯一的目标就是不断预测下一个 Token，通过数万亿次的练习
逐渐学会语言的规律。
```

### 2.2 数据准备

预训练数据是整个训练中最重要但也最「脏」的部分：

```
                         原始数据总量
                     (互联网、书籍、代码...)
                              │
                              ▼
                    ┌──────────────────┐
                    │   ① 数据清洗       │
                    │  去重、去HTML、     │
                    │  去个人信息、       │
                    │  过滤低质量内容     │
                    └────────┬─────────┘
                             │
                             ▼
                    ┌──────────────────┐
                    │   ② 数据配比       │
                    │  网页: 60%         │
                    │  书籍: 15%         │
                    │  代码: 15%         │
                    │  学术: 5%          │
                    │  其他: 5%          │
                    └────────┬─────────┘
                             │
                             ▼
                    ┌──────────────────┐
                    │   ③ Tokenization  │
                    │  BPE / Sentence-  │
                    │  Piece 分词       │
                    └────────┬─────────┘
                             │
                             ▼
                        训练就绪数据
                     (数万亿 Tokens)
```

> [!warning] 数据质量 > 数据数量
> Chinchilla 定律指出：在给定计算预算下，存在最优的模型大小和数据量配比。
> 但实践中发现，**高质量的小数据集可以胜过低质量的大数据集**。

### 2.3 常见的预训练数据来源

| 数据来源 | 规模估计 | 用途 | 注意事项 |
|----------|---------|------|----------|
| Common Crawl | PB 级 | 网页文本（主力） | 需要大量清洗 |
| Wikipedia | ~20GB | 结构化知识 | 高质量但有限 |
| Books3 / Books | ~100GB | 长篇连贯文本 | 版权争议 |
| GitHub | ~TB 级 | 代码能力 | 许可证问题 |
| arXiv / PubMed | ~100GB | 学术推理 | STEM 能力 |
| Reddit / 论坛 | ~100GB | 对话/口语化语言 | 偏见和有害内容 |

### 2.4 训练配置

以 Llama 3 8B 为例：

```
配置项                     值                    说明
───────────────────────────────────────────────────────
参数量                    8B                    ~80 亿
层数                      32                    每层一个 Attention + FFN
隐藏维度                  4096
注意力头数                32                    每个 Head: 128 维
词表大小                  128,000               TikToken-based
上下文长度                8K tokens             (训练时)
优化器                    AdamW                 β₁=0.9, β₂=0.95
学习率                    3e-4                  余弦衰减
Batch Size               4M tokens             全局批量
训练数据总量              ~15T tokens           约 2200 万本书
训练时长                  ~1.3M GPU-hours       (H100-80GB)
GPU 数量                  16,000 H100           约 80 天
```

### 2.5 训练损失曲线

```
Loss
  │
  │⡤
  │ ⠀⡤
  │   ⠀⡤
  │     ⠀⢤
  │       ⠀⢤
  │         ⠀⢤
  │           ⠀⢤
  │             ⠀⢤
  │               ⠀⠉⠉⠉⠉ ╮ ← 收敛区域（缓慢下降）
  │
  └────────────────────────────► Steps

  训练初期 (前 20%): Loss 快速下降，模型学会基础语法
  训练中期 (20%-60%): 缓慢但稳定下降，学会更多知识
  训练后期 (60%+): 逼近收敛，继续提升但越来越「贵」
```

## 3. 第二阶段：监督微调（SFT）

### 3.1 为什么需要 SFT？

预训练完的 Base Model 只会「续写」文本，不会「对话」：

```
Base Model 的行为:
  输入:  "请介绍一下北京"
  输出:  "。北京市是中国的首都，也是中国...请介绍一下上海"
         ↑ Base Model 把 prompt 当成文章的一部分来续写！

Chat Model 的行为 (SFT 后):
  输入:  "请介绍一下北京"
  输出:  "北京是中华人民共和国的首都，位于华北平原北部..."
         ↑ SFT 后的模型理解了这是问答格式
```

### 3.2 SFT 数据格式

SFT 数据由高质量问答对组成：

```
典型 SFT 数据:
┌────────────────────────────────────────────┐
│                                            │
│  {                                         │
│    "messages": [                           │
│      {"role": "system", "content": "你是...│
│      {"role": "user", "content": "请解释...│
│      {"role": "assistant", "content": "量...│
│    ]                                       │
│  }                                         │
│                                            │
│  通常需要 10K~100K 条高质量对话             │
└────────────────────────────────────────────┘
```

### 3.3 SFT 数据来源

```
SFT 数据来源:
┌──────────────────────────────────────┐
│ 1. 人工标注（最贵但质量最高）          │
│    → 聘请领域专家写回答               │
│                                      │
│ 2. 合成数据（主流方式）               │
│    → 用更强的模型 (GPT-4) 生成回答    │
│    → 成本低，可大规模生产             │
│                                      │
│ 3. 社区数据                           │
│    → ShareGPT、OpenAssistant 等      │
│    → 多样性好，质量参差不齐            │
│                                      │
│ 4. 自我指导 (Self-Instruct)           │
│    → 让模型自己生成问题和回答          │
│    → 用种子任务扩展                   │
└──────────────────────────────────────┘
```

## 4. 第三阶段：对齐训练（Alignment）

即使模型会对话了，它的回答仍可能有害、有偏见、或不按指令来。对齐训练让模型「学会做人」。

### 4.1 RLHF（人类反馈强化学习）

RLHF 是最经典的对齐方案（由 InstructGPT / ChatGPT 发扬光大）：

```
RLHF 三步走:

┌──────────────────────────────────────────────────────────────┐
│ Step 1: 收集人类偏好数据                                       │
│                                                              │
│  给标注员展示:                                                │
│    Prompt: "如何制作炸弹？"                                    │
│    Response A: "我无法提供制作炸弹的指导..." ← 更好            │
│    Response B: "首先你需要硝酸钾..."       ← 更差             │
│    → 标注员选择 A > B                                         │
│                                                              │
│  收集大量比较对 → 偏好数据集                                   │
└──────────────────────┬───────────────────────────────────────┘
                       ▼
┌──────────────────────────────────────────────────────────────┐
│ Step 2: 训练奖励模型 (Reward Model)                            │
│                                                              │
│  RM(prompt, response) → 一个分数（这个回答有多好？）           │
│                                                              │
│  基于 SFT 模型，将最后的输出层替换为回归头                      │
│  训练目标: 让好回答的分数 > 差回答的分数                        │
└──────────────────────┬───────────────────────────────────────┘
                       ▼
┌──────────────────────────────────────────────────────────────┐
│ Step 3: PPO 强化学习优化                                       │
│                                                              │
│  For each prompt:                                            │
│    模型生成回答 → RM 打分 → 用 PPO 更新参数 → 让分数越来越高     │
│                                                              │
│  同时加入 KL 惩罚: 不要让模型偏离 SFT 版本太远                  │
│  (防止「只为了高分而胡言乱语」)                                 │
└──────────────────────────────────────────────────────────────┘
```

### 4.2 DPO（直接偏好优化）

DPO 是 RLHF 的简化版，去掉了奖励模型：

```
RLHF 的问题:
  - 需要单独训练一个奖励模型
  - PPO 训练不稳定，超参数敏感
  - 流程复杂，工程量大

DPO 的改进:
  直接使用偏好数据来优化模型，一步到位
  
  Loss_DPO = -log σ( β × (log π(y_good|x) - log π(y_bad|x)) )
  
  核心直觉: 让好回答的生成概率 > 坏回答的生成概率

  DPO 不需要:
  ┌────────────────────────┐
  │ ✗ 奖励模型              │
  │ ✗ PPO 强化学习          │
  │ ✗ 复杂的超参数调优       │
  └────────────────────────┘

  目前 DPO 及其变体 (ORPO, SimPO) 已基本取代 RLHF 成为主流
```

### 4.3 RLHF vs DPO 对比

| 维度 | RLHF | DPO |
|------|------|-----|
| 训练流程 | 3 步（SFT → RM → PPO） | 1 步 |
| 需要奖励模型 | 是 | 否 |
| 训练稳定性 | 较不稳定 | 稳定 |
| 计算开销 | 大 | 小 |
| 效果上限 | 高（理论） | 接近 RLHF |
| 主流程度 | 早期标准 | 当前主流 |

## 5. 分布式训练概述

大模型的参数量远超单张 GPU 的内存，必须分布式训练：

```
并行策略组合:

┌──────────────────────────────────────────────┐
│ 数据并行 (Data Parallelism)                    │
│   每张 GPU 持有完整模型副本，处理不同数据批次    │
│   梯度在 GPU 间同步 (AllReduce)                │
├──────────────────────────────────────────────┤
│ 张量并行 (Tensor Parallelism)                 │
│   将单层权重切分到多张 GPU                      │
│   适合单机多卡场景 (NVLink 高速互联)            │
├──────────────────────────────────────────────┤
│ 流水线并行 (Pipeline Parallelism)             │
│   不同层放在不同 GPU 上                         │
│   按批次流水线执行，减少空闲时间                 │
├──────────────────────────────────────────────┤
│ 序列并行 (Sequence Parallelism)               │
│   将长序列切分到多卡                            │
│   配合 Ring Attention 使用                     │
└──────────────────────────────────────────────┘

实际训练中通常是 3D 并行:
  数据并行 × 张量并行 × 流水线并行
```

## 6. 评估体系

### 6.1 常用评测基准

| 基准 | 测试方向 | 说明 |
|------|---------|------|
| MMLU | 多学科知识 | 57 个学科的多选题，衡量知识广度 |
| HumanEval | 代码生成 | 根据 docstring 补全函数 |
| GSM8K | 数学推理 | 小学数学应用题 |
| MATH | 数学推理 | 竞赛级数学题 |
| HellaSwag | 常识推理 | 选择最合理的句子续写 |
| TruthfulQA | 真实性 | 检测模型是否会产生错误信念 |
| MT-Bench | 多轮对话 | GPT-4 作为裁判评估对话质量 |
| Arena Elo | 综合能力 | Chatbot Arena 人类投票排名 |

### 6.2 评测策略

```
评测层次:
  单点评测 → 多基准综合 → 人类评估 → 真实场景 A/B Test

常见陷阱:
  - 数据污染: 评测数据出现在训练集中 → 分数虚高
  - 评测偏差: 不同评测框架结果不一致
  - 过拟合: 针对评测基准特化训练 → 泛化能力差
```

## 7. 训练成本估算

以 Llama 3 8B 为例（使用 15T tokens 数据训练）：

```
计算量 (FLOPs):
  约 1.3 × 10²⁴ FLOPs (1.3 YottaFLOPs)

训练成本:
  H100-80GB × 16,000 张 × ~80 天
  ≈ 1,280,000 GPU-天数
  ≈ 30,720,000 GPU-小时

粗略成本 (云 GPU $2/GPU-hour):
  ≈ $61,000,000 (约 6100 万美元)

  加上: 数据采集和清洗、工程师薪资、消融实验 → 实际成本更高
```

> [!warning] 这还只是 8B 级别
> GPT-4 的训练成本估计在 1-2 亿美元。越大的模型越贵，但小模型 + 多数据可能更优（Chinchilla 最优）。

## 关键概念速查表

| 中文概念 | 英文 | 一句话解释 |
|----------|------|-----------|
| 预训练 | Pretraining | 在海量语料上通过预测下一个 Token 学习语言基础 |
| 监督微调 | SFT | 使用高质量问答对教模型学会对话格式 |
| RLHF | Reinforcement Learning from Human Feedback | 用人类偏好训练奖励模型，再用强化学习优化 |
| DPO | Direct Preference Optimization | 直接使用偏好数据优化，不需要奖励模型 |
| 基座模型 | Base Model | 预训练完成但未经过 SFT 的原始模型 |
| 聊天模型 | Chat Model | 经过 SFT + 对齐训练、可以直接对话的模型 |
| Tokenization | Tokenization | 将文本切分为模型可处理的 Token 序列 |
| 奖励模型 | Reward Model | 在 RLHF 中用于打分评估回答质量的模型 |
| KL 惩罚 | KL Penalty | 防止强化学习后模型与原始策略偏差过大 |
| 分布式训练 | Distributed Training | 使用多张 GPU/多台机器协同训练大模型 |
| 评估基准 | Benchmark | 用于衡量模型能力的标准化测试集 |
