# 大模型推理与优化

## 概述

训练好的模型最终要用来「干活」——这个过程叫推理（Inference）。推理的核心挑战是：**如何在有限的硬件上，让巨大的模型跑得快、跑得省**。本文覆盖推理的完整流程，以及 KV Cache、量化、投机采样等核心优化技术。

> [!note] 前置知识
> 建议先阅读 [[大模型技术原理]] 了解 Transformer 推理过程，以及 [[大模型训练流程]] 了解训练和推理的区别。

## 1. 训练 vs 推理 — 关键区别

| 维度 | 训练 (Training) | 推理 (Inference) |
|------|----------------|-----------------|
| **方向** | 前向 + 反向（梯度） | 仅前向 |
| **并行方式** | 数据处理同一 prompt | 逐个 Token 生成（串行） |
| **硬件瓶颈** | 算力（FLOPS） | 内存带宽 |
| **精度要求** | FP16/BF16 | INT4/INT8 可能足够 |
| **Batch** | 大 Batch (百万 Token 级) | 小 Batch（通常为 1） |
| **耗时** | 数周 | 毫秒~秒级/请求 |
| **目标** | 降低 Loss | 降低延迟 + 提高吞吐 |

```
为什么推理是「内存瓶颈」？

自回归生成是串行的:
  输入 "今天" → 生成 "天气" → [今天, 天气] → 生成 "真" → ...
  
每一步需要:
  1. 加载全部模型权重 (内存 I/O)
  2. 做一次前向计算 (可以很快)
  
大部分时间花在把权重从显存搬运到计算单元上。
```

## 2. 自回归生成流程

```
                     ┌───────────┐
                     │  开始生成   │
                     └─────┬─────┘
                           │
                           ▼
                  ┌────────────────┐
                  │ 输入 Prompt     │
                  │ "今天天气"      │
                  └───────┬────────┘
                          │
                ┌─────────▼─────────┐
                │  Forward Pass     │◄── 处理所有 prompt tokens
                │  (Prefill 阶段)   │    并行处理，计算密集型
                └─────────┬─────────┘
                          │
                ┌─────────▼─────────┐
                │  采样 1 个 Token  │
                │  生成: "真"       │
                └─────────┬─────────┘
                          │
                ┌─────────▼─────────┐
                │  Forward Pass     │◄── 只处理一个新 token
                │  (Decode 阶段)    │    串行处理，内存密集型
                └─────────┬─────────┘
                          │
                ┌─────────▼─────────┐
                │  生成结束？        │
                │  (EOS / max_len)  │
                └────┬─────────┬────┘
                     │ 否      │ 是
                     ▼         ▼
              ┌──────────┐  ┌──────────┐
              │ 继续生成   │  │  返回结果  │
              └──────────┘  └──────────┘

两个阶段:
  Prefill (预填充): 一次性处理所有输入 Token → 可以并行
  Decode (解码):    每次生成一个 Token → 必须串行 → 速度瓶颈
```

## 3. KV Cache — 推理加速的核心

### 3.1 问题：重复计算

```
不用 KV Cache (每一步都重新计算):

  生成 "真":  处理 ["今天", "天气"]          → 2 个 Token
  生成 "好":  处理 ["今天", "天气", "真"]    → 3 个 Token
  生成 "啊":  处理 ["今天", "天气", "真", "好"] → 4 个 Token

每一步都在重复计算之前所有 Token 的 K 和 V！
```

### 3.2 方案：缓存 K 和 V

```
KV Cache 原理:

  生成 "真":  计算 ["今天", "天气"] 的 K,V → 存入缓存
             只用最新 Token 的 Q 做 Attention → 生成 "真"
             
  生成 "好":  从缓存读取之前所有 K,V
             只计算 "真" 的 K,V → 追加到缓存
             用 "好" 的 Q 做 Attention → 生成 "好"
             
  每个 Decode 步骤只计算 1 个 Token 的 K,V！

┌──────────────────────────────────────────────┐
│ 复杂度变化:                                    │
│   无 Cache: O(n²)  (n = 序列长度)              │
│   有 Cache: O(n)   (每一步只算一个新 Token)     │
│                                              │
│ 但代价是: 需要大量显存来存储 KV Cache           │
└──────────────────────────────────────────────┘
```

### 3.3 KV Cache 内存估算

```
以 Llama 3 8B 为例:

每层需存储:
  - K: batch × num_heads × seq_len × head_dim
  - V: batch × num_heads × seq_len × head_dim

单层 KV Cache 大小 (1 个请求, 4096 tokens):
  2 × 1 × 32 × 4096 × 128 × 2 bytes (FP16)
  = 67,108,864 bytes ≈ 64 MB

32 层总共: 64 MB × 32 ≈ 2 GB

8 个并发请求 × 4096 tokens:
  ≈ 16 GB  — 仅 KV Cache 就占满了一张 RTX 4090！
```

> [!warning] KV Cache 是大模型推理的「显存黑洞」
> 上下文越长，并发越多，KV Cache 增长越快。这是长上下文推理的核心瓶颈。

### 3.4 KV Cache 优化技术

| 技术 | 原理 | 效果 |
|------|------|------|
| **GQA (分组查询注意力)** | 多个 Q Head 共享 KV Head | KV Cache 缩减 4-8 倍 |
| **MQA (多查询注意力)** | 所有 Head 共享 KV | KV Cache 缩减 32 倍（但质量下降） |
| **Multi-Query Latent** | DeepSeek 的 MLA 技术 | KV 压缩到极致的潜空间表示 |
| **PagedAttention** | 将 KV Cache 分页管理（vLLM 使用） | 减少碎片，提高利用率 |
| **量化 KV Cache** | 将 KV Cache 从 FP16 量化到 INT8/FP8 | 内存减半 |
| **滑动窗口** | 只保留最近 N 个 Token 的 KV Cache | 适用于流式场景 |

## 4. 模型量化（Quantization）

### 4.1 什么是量化？

将模型的权重（和激活值）从高精度格式转换为低精度格式：

```
FP16 (半精度):    每个权重占 16 bits (2 bytes)
INT8 (8位整数):    每个权重占 8 bits (1 byte)    → 体积减半
INT4 (4位整数):    每个权重占 4 bits (0.5 byte)  → 体积为 1/4

Llama 3 8B (FP16):
  8B × 2 bytes = 16 GB 显存

Llama 3 8B (INT4):
  8B × 0.5 bytes = 4 GB 显存  → 可以在 RTX 3070 (8GB) 上运行！
```

### 4.2 量化方法对比

| 方法 | 类型 | 精度损失 | 加速比 | 代表工具 |
|------|------|---------|--------|----------|
| GPTQ | 训练后量化 | 较小 | 2-4× | AutoGPTQ |
| AWQ | 训练后量化 | 小 | 2-4× | AutoAWQ |
| GGUF | 文件格式 + 量化 | 取决于位数 | 1-3× | llama.cpp |
| BitsAndBytes | 在线量化 | 小 | 无加速（仅减内存） | transformers |
| FP8 | 原生精度 | 极小 | 4× (硬件支持) | H100/B200 |

```
量化方法的权衡:

  质量  ▲
       │  ● FP16 (原始)
       │    ● INT8 (几乎无损)
       │       ● INT4 (轻度退化)
       │          ● INT2 (明显退化)
       └──────────────────────► 体积缩减
```

### 4.3 GGUF 格式与 llama.cpp 生态

```
GGUF 格式是本地推理的事实标准:

  ┌──────────────────────────────────────────┐
  │ llama.cpp / ollama / LM Studio 等工具     │
  │ 都使用 GGUF 格式的量化模型                 │
  │                                          │
  │ 常见量化级别:                              │
  │   Q4_K_M  — 最佳性价比（推荐）             │
  │   Q5_K_M  — 质量稍好，体积稍大             │
  │   Q8_0    — 接近无损                       │
  │   Q2_K    — 极限压缩                       │
  │                                          │
  │ 命名规则: Q{bits}_{算法}_{大小}            │
  │   K = k-quant (最佳量化策略)               │
  │   M/S = Medium/Small (block 大小)         │
  └──────────────────────────────────────────┘
```

## 5. 推理优化技术全景

### 5.1 Flash Attention

```
标准 Attention 的问题:
  计算 Attention Matrix (n×n) → 存储到 HBM (显存) → 读取做 Softmax

  这个写-读操作是推理的主要瓶颈（内存带宽）

Flash Attention 的解决方案:
  用分块计算 (Tiling) + 在线 Softmax，避免将完整 Attention Matrix
  写入 HBM。所有计算在 SRAM (L1 Cache) 内完成。

  HBM (大但慢)             SRAM (小但快)
  ┌──────────┐            ┌──────────────┐
  │ Q,K,V    │────读取───►│ Flash Attn   │
  │ (各 2GB) │◄───写回───│ 分块计算      │
  │          │            │ 不需要存储    │
  │ Output   │            │ 中间矩阵      │
  └──────────┘            └──────────────┘

  加速: 2-4×，几乎不损失精度
  现已成标准配置 (PyTorch 2.0+ 内置支持)
```

### 5.2 投机采样（Speculative Decoding）

```
问题: 自回归生成每次只能生成 1 个 Token

投机采样的解决方案:
  用一个小模型（Draft Model）快速生成多个候选 Token，
  然后用大模型（Target Model）一次性验证它们。

  ┌──────────┐    候选 Token (5 个)     ┌──────────┐
  │ 小模型    │ ──────────────────────► │ 大模型    │
  │ (快速)    │                         │ (验证)    │
  │ 13M 参数  │ ◄────────────────────── │          │
  └──────────┘    接受/拒绝 + 修正       └──────────┘

  每次验证 = 1 次前向 → 可能接受 3-5 个 Token
  加速: 2-3×，完全无损（大模型保证输出质量）
```

### 5.3 技术汇总

| 优化技术 | 优化目标 | 加速效果 | 精度影响 |
|----------|---------|---------|---------|
| KV Cache | 避免重复计算 | 基础技术，必须 | 无 |
| GQA/MQA | KV Cache 内存 | 减少 4-32× 显存 | 极小 |
| Flash Attention | 内存带宽 | 2-4× | 无 |
| 量化 (INT8) | 模型体积 + 内存带宽 | 2-3× | 极小 |
| 量化 (INT4) | 模型体积 + 内存带宽 | 4-6× | 轻度 |
| 投机采样 | 计算效率 | 2-3× | 无 |
| PagedAttention | 显存利用率 | 吞吐提升 2-4× | 无 |
| Continuous Batching | 吞吐量 | 提升 10×+ | 无 |
| Tensor Parallelism | 多卡加速 | N 卡 ≈ N× | 无 |

```
┌─────────────────────────────────────────────────────────┐

└─────────────────────────────────────────────────────────┘
```

### 6.2 典型部署方案

```
场景 A: 个人本地使用
  方案: Ollama + Open WebUI
  硬件: RTX 4090 (24GB) 或 Mac M2 Ultra (64GB)
  模型: Llama 3 8B Q4_K_M (~6GB) 或 Qwen 2.5 7B

场景 B: 小团队 API 服务
  方案: vLLM + FastAPI
  硬件: 2× A100 (40GB/80GB)
  模型: Qwen 2.5 72B AWQ 量化

场景 C: 生产环境高并发
  方案: vLLM / TensorRT-LLM + K8s
  硬件: 8× H100 集群
  模型: 多模型混合部署 + 自动扩缩容
```

### 6.3 Ollama 快速部署示例

```bash
# 安装 Ollama
curl -fsSL https://ollama.com/install.sh | sh

# 拉取模型
ollama pull qwen2.5:7b          # Qwen 2.5 7B
ollama pull llama3.1:8b         # Llama 3.1 8B

# Modelfile 自定义
cat > Modelfile << EOF
FROM qwen2.5:7b
PARAMETER temperature 0.7
PARAMETER top_p 0.9
SYSTEM "你是一个专业的技术助手"
EOF

ollama create my-assistant -f Modelfile

# 运行
ollama run my-assistant
```

## 7. 幻觉问题与缓解

### 7.1 什么是幻觉？

```
幻觉 (Hallucination): 模型生成的内容看似可信，但与事实不符。

类型:
  事实性幻觉: "鲁迅的代表作《三体》..." (鲁迅没写三体)
  忠实性幻觉: 偏离用户指令或上下文
  逻辑性幻觉: 前后矛盾、"1+1=3"
```

### 7.2 缓解策略

| 策略 | 原理 | 效果 |
|------|------|------|
| **RAG (检索增强生成)** | 先检索相关文档，再让模型基于文档回答 | 大幅减少幻觉，是目前最有效的手段 |
| **降低 Temperature** | 让模型输出更确定 | 减少随机性幻觉 |
| **System Prompt 约束** | 明确要求「不知道就说不知道」 | 简单有效 |
| **CoT / 推理链** | 让模型分步骤推理 | 提高推理准确性 |
| **工具调用** | 计算/查询交给外部工具 | 消除「1+1=3」类错误 |
| **COT + 验证** | 让模型自己验证自己的输出 | 进一步降低错误率 |
